Troubleshooting Docker

This runbook documents Docker operational tasks performed on a Linux host running Ubuntu, covering installation verification, permission management, Docker Compose v2 usage, Postgres 18 volume migration, and container troubleshooting. All steps and outputs below are drawn from a real deployment session of the Peppermint helpdesk application.

1. Resolving Docker Socket Permission Errors

  • Running any Docker command as a non-root user produced: 
      $ docker ps
  permission denied while trying to connect to the Docker daemon socket at unix:///var/run/docker.sock
Cause

The Docker daemon socket is owned by the docker group. A user must be a member of this group to communicate with the daemon without elevated privileges.

Resolution — Add User to the docker Group
  • Step 1: Add the current user to the docker group: 
      $ sudo usermod -aG docker $USER
  • Step 2: Apply the group change, log out and back in, or activate immediately: 
      $ newgrp docker
  • Step 3: Verify membership, the word docker must appear in the output: 
      $ groups

Members of the docker group have root-equivalent host access. Only add trusted users. In production, consider rootless Docker.

Immediate Workaround, Prefix commands with sudo as a temporary measure while group changes propagate sudo docker ps

Suppressing the ‘version’ Attribute Warning
  • The following warning appeared on every compose command: 
      WARN[0000] /home/sysadmin/docker/peppermint/docker-compose.yaml: the attribute `version` is obsolete, it will be ignored, please remove it to avoid potential   confusion
  • Resolution: remove the top-level version key from docker-compose.yaml entirely. Compose v2 detects the file format automatically. 
     # Before
 version: '3.8'
 services:
 ...

     # After — remove the version line completely
 services:
 ...
Postgres Container Crash Loop
  • After bringing the stack up, docker ps showed the Postgres container in a restart loop: 
      $ docker ps

  CONTAINER ID   IMAGE                          STATUS                         NAMES
  d605aa21360f   pepperlabs/peppermint:latest   Up 3 minutes                   peppermint
  33bb5e6a1fd1   postgres:latest                Restarting (1) 9 seconds ago   peppermint_postgres
Root Cause — Postgres 18 Changed the Data Directory Structure
  • Running docker logs peppermint_postgres revealed: 
      $ docker logs peppermint_postgres

  Error: in 18+, these Docker images are configured to store database data in a
     format which is compatible with "pg_ctlcluster" (specifically, using
     major-version-specific directory names).  This better reflects how
     PostgreSQL itself works, and how upgrades are to be performed.
     See also https://github.com/docker-library/postgres/pull/1259
  Counter to that, there appears to be PostgreSQL data in:/var/lib/postgresql/data (unused mount/volume)
  This is usually the result of upgrading the Docker image without upgrading the underlying database using "pg_upgrade" (which requires both versions).
  The suggested container configuration for 18+ is to place a single mount at /var/lib/postgresql which will then place PostgreSQL data in a subdirectory, allowing usage of "pg_upgrade --link" without mount point boundary issues.
  • Postgres 18 no longer uses /var/lib/postgresql/data as the mount target. It now expects the mount at /var/lib/postgresql and manages its own versioned subdirectory beneath it.
Resolution (Postgres 18 Volume Path Migration)
  • Step 1 — Tear down the stack and remove the old volume 
      $ docker compose down -v

  [+] down 4/4
  ✔ Container peppermint          Removed    4.4s
  ✔ Container peppermint_postgres Removed    3.7s
  ✔ Volume peppermint_pgdata      Removed    0.1s
  ✔ Network peppermint_default    Removed
  • Step 2 — Update the volume mount in docker-compose.yaml 
      # Before (Postgres 17 and earlier)
  volumes:
     - peppermint_pgdata:/var/lib/postgresql/data

      # After (Postgres 18+)
  volumes:
     - peppermint_pgdata:/var/lib/postgresql

Change the Postgres volume mount from the old path to the new Postgres 18 path:

  • Step 3 — Bring the stack back up 
      $ docker compose up -d

Postgres 18 will initialise a fresh database in the correct versioned subdirectory. Both containers should reach a healthy running state.

Never use postgres:latest in production. Pin a specific major version (e.g. postgres:17) so that a major version bump cannot silently break your stack on the next docker compose pull.

2. Common Error Reference

  • This table captures some errors I ran across whilst configuring my homelab.
Error Cause Resolution
docker-compose: command not found Legacy binary removed in v23+ Use "docker compose" (with a space)
permission denied …/docker.sock User not in docker group sudo usermod -aG docker $USER then re-login
unable to get image … permission denied User not in docker group Same as above, or prefix command with sudo
version attribute is obsolete (WARN) Outdated Compose file Remove the version: line from docker-compose.yaml
Cannot connect to the Docker daemon Docker service not running sudo systemctl start docker
Restarting (1) … postgres container Volume path mismatch (Postgres 18) Mount at /var/lib/postgresql not …/data; run down -v first
data appears to be PostgreSQL data in /var/lib/postgresql/data Postgres 18 changed data directory layout Update volume mount and wipe old volume with down -v
