Working with Docker

The aim of this article is to cover a basic session on how to work with docker, these are basic entry level tasks to help you get comfortable. However, I do encourage working and building different projects which will expose you to troubleshooting scenarios allowing you to learn more advanced concepts.

Prerequisites

  • A very very basic understanding of the Linux command line
  • Some knowledge of docker, here is a good primer

Why Not Use apt install docker.io

Ubuntu’s default apt repositories include a package called docker.io. This should be avoided for production use because it is maintained by the Ubuntu team, not Docker, and is typically several major versions behind the official Docker release.

The correct approach is to add Docker’s official repository and install from there. This gives you the latest stable release, the official CLI, BuildKit support, and the Compose plugin all of which are needed for running Supabase (I’m referencing Supabase here but this applies to any application, my current docker setup is built around Supabase).

Installation Steps

As a disclaimer, alot of these commands and infromation are readily available on the docker official website. My coverage is simply for my own notes and/or if you choose to follow this guide verbatim as most commands and outputs are pulled from my working system.

Step 1 - Add Docker’s official GPG key

This key is used to verify packages downloaded from Docker’s repository are authentic and have not been tampered with.

install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

Step 2 — Add the Docker apt repository

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
   https://download.docker.com/linux/ubuntu \
   $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
   sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

The $(. /etc/os-release && echo “$VERSION_CODENAME”) part automatically detects the Ubuntu release codename. On Ubuntu 24.04 this resolves to ‘noble’. This means the same command works across Ubuntu versions without modification.

Step 3 — Install Docker packages

Update apt to include the new Docker repository, then install all required Docker components:

sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

docker-ce Docker Engine — the core container runtime
docker-ce-cli Docker CLI — the docker command
containerd.io Container runtime used by Docker Engine
docker-buildx-plugin Extended image build capabilities
docker-compose-plugin Enables docker compose commands

Step 4 — Add current user to the docker group and Verify the Docker installation

sudo usermod -aG docker $USER
docker ps
    CONTAINER ID   IMAGE     COMMAND   CREATED   STATUS    PORTS     NAMES

Adding yourself/current user to user group will address any permission errors. Basically the Docker daemon runs as root and exposes its API via a Unix socket at /var/run/docker.sock. By default only root and members of the docker group can read/write that socket. Adding yourself to the docker group grants that access permanently.

Basically, the -p flag maps a port on your host machine to a port inside the container. The format is always host:container. To explain containers run in their own network namespace they’re isolated from your host network by default. Port 80 inside the container is completely invisible to the outside world unless you explicitly punch a hole through with -p. Without it, nothing can reach your containerized app, even from the same machine. Also you only expose ports for services that genuinely need to be reachable from outside. containers on the same Docker network talk to each other directly by container name without exposing anything to the host.

From the above we can see that our image is being pulled from a different registry name: gcr.io. You can also create and manage your own container images. The process is fairly straight forward once you understand your requirements and dependencies. Let’s build a quick ngnix container with some customizaion.

cat > Dockerfile <<EOF
> FROM nginx
> COPY . /usr/share/nginx/html
> EXPOSE 80/tcp
> CMD ["nginx", "-g daemon off;"]
> EOF

Images, containers, volumes, or custom configuration files on your host aren’t automatically removed. To delete all images, containers, and volumes, run

sudo rm -rf /var/lib/docker
sudo rm -rf /var/lib/containerd
