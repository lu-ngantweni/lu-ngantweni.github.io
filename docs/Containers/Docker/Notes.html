1. Container Lifecycle

Replace the “testapp” with the name of your container or container ID, these details can be obtained from the output of the docker ps command.

Basic Operations

Command Description
docker start testapp Start a stopped container
docker stop testapp Gracefully stop a running container (SIGTERM, then SIGKILL after 10s)
docker restart testapp Stop then start a container
docker kill testapp Immediately kill a container (SIGKILL)
docker rm testapp Remove a stopped container
docker rm -f testapp Force remove a running container
docker run -d --name testapp --restart unless-stopped image:tag Run container in background with auto-restart policy

Always set a –restart policy (unless-stopped or on-failure) for long-running services so containers recover after host reboots.

Inspect Running Containers

Command Description
docker ps Start a stopped container
docker ps -a Gracefully stop a running container (SIGTERM, then SIGKILL after 10s)
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}' Custom output format
docker inspect testapp Full JSON metadata for a container
docker stats Live CPU / memory / network / disk I/O
docker top testapp Show running processes inside a container
docker port testapp Show port mappings

2. Image Management

Command Description
docker images List local images
docker pull image:tag Pull an image from a registry
docker build -t name:tag . Build an image from a Dockerfile in current directory
docker rmi image:tag Remove a local image
docker rmi $(docker images -q --filter dangling=true) Remove all dangling (untagged) images
docker image prune -a Remove all unused images (not referenced by any container)
docker tag src:tag dst:tag Tag an image with a new name
docker push image:tag Push image to a registry
docker save image:tag | gzip > image.tar.gz Export image to a compressed tar
docker load < image.tar.gz Import a previously saved image

Always pin image versions in production (e.g. postgres:17, not postgres:latest). Latest can silently pull a major version bump with breaking changes on the next pull.

3. Logs & Debugging

Working with logs

Command Description
docker logs testapp Print all logs for a container
docker logs -f testapp Tail logs in real time (Ctrl+C to exit)
docker logs --tail 100 testapp Show last 100 lines only
docker logs --since 1h testapp Show logs from the past hour
docker logs --timestamps testapp Include timestamps in output

Exec & Shell Access

Command Description
docker exec -it testapp bash Open an interactive bash shell inside a running container
docker exec -it testapp sh Use sh if bash is not available (Alpine-based images)
docker exec testapp cat /etc/hosts Run a single command without an interactive shell
docker exec -u root testapp bash Shell in as root user

Copying Files

Command Description
docker cp testapp:/path/in/container ./local Copy a file out of a container
docker cp ./local testapp:/path/in/container Copy a file into a container

4. Docker Compose Commands

All commands use “docker compose” (space, no hyphen). The legacy docker-compose binary is not installed on Docker Engine v23+.

Command Description
docker compose up -d Start all services in detached mode
docker compose up -d --build Rebuild images before starting
docker compose up -d <service> Start a single named service
docker compose down Stop and remove containers and networks
docker compose down -v Also remove named volumes (destructive — wipes DB data)
docker compose restart Restart all services
docker compose restart <service> Restart a single service
docker compose stop Stop containers without removing them
docker compose start Start previously stopped containers
docker compose pull Pull latest images for all services
docker compose ps List service status
docker compose logs -f Tail logs for all services
docker compose logs -f <service> Tail logs for one service
docker compose exec <service> bash Shell into a running service container
docker compose config Validate and display the resolved compose config

Compose File Best Practices

  • Never use image: postgres:latest for databases — always pin a major version such as postgres:17.
  • For Postgres 18+, mount volumes at /var/lib/postgresql (not /var/lib/postgresql/data).
  • Remove the top-level version: key it is obsolete in Compose v2 and generates a warning.
  • Use named volumes (not bind mounts) for database data to avoid host permission issues.
  • Set restart: unless-stopped on all long-running services.
  • Use .env files for secrets never hardcode passwords in docker-compose.yaml.

5. Volumes & Networks

Volumes

Command Description
docker volume ls List all volumes
docker volume inspect <name> Show volume metadata and mount path
docker volume rm <name> Remove a volume (must not be in use)
docker volume prune Remove all unused volumes
docker run -v myvolume:/data image Mount a named volume into a container

docker volume prune and docker compose down -v are destructive. They permanently delete volume data including database files. Always confirm before running.

Networks

Command Description
docker network ls List all networks
docker network inspect <name> Show network details and connected containers
docker network create mynet Create a custom bridge network
docker network connect mynet <container> Connect a running container to a network
docker network disconnect mynet <container> Disconnect a container from a network
docker network prune Remove unused networks

6. System Cleanup

Command Description
docker system df Show disk usage by images, containers, volumes, cache
docker system prune Remove stopped containers, unused networks, dangling images and build cache
docker system prune -a Also remove all unused images (not just dangling ones)
docker system prune -a -v Also remove unused volumes — most aggressive, use with caution
docker image prune Remove only dangling images
docker container prune Remove only stopped containers
docker volume prune Remove only unused volumes
docker network prune Remove only unused networks

Run docker system df regularly to monitor disk usage. Build cache can accumulate quickly on CI hosts.

7. Permissions & the Docker Group

Command Description
sudo usermod -aG docker $USER Add current user to the docker group
newgrp docker Activate group change without re-logging
groups Verify current user group memberships
ls -la /var/run/docker.sock Check socket ownership and permissions
sudo systemctl status docker Check whether the Docker daemon is running
sudo systemctl restart docker Restart the Docker daemon

The docker group grants root-equivalent access to the host. Only add trusted users. In production environments consider running Docker in rootless mode instead.

8. Troubleshooting Checklist

Container won’t start / keeps restarting

  • Run docker logs to read the error output.
  • Run docker inspect and check the State section for exit codes.
  • Verify all required environment variables are set in docker-compose.yaml.
  • Check volume mount paths — especially for Postgres 18+ (see Section 4).
  • Confirm the image tag exists: docker pull .

Cannot connect to Docker daemon

  • Check the daemon is running: sudo systemctl status docker.
  • Check user permissions: groups — docker must appear.
  • If missing from groups: sudo usermod -aG docker $USER then log out and back in.
  • As an immediate workaround, prefix the command with sudo.

Port already in use

  • Identify what is using the port: sudo ss -tlnp | grep <port>
  • Stop the conflicting process or change the host port in docker-compose.yaml.

Out of disk space

  • Check usage: docker system df
  • Remove dangling images and stopped containers: docker system prune
  • For a full clean (caution — removes all unused resources): docker system prune -a -v

General diagnostic sequence

  • docker ps -a see status of all containers
  • docker logs <container> read the error
  • docker inspect <container> check full config and state
  • docker compose config validate the compose file
  • sudo systemctl status docker check the daemon itself

9. Best Practices Summary

  • Pin image versions — never use :latest for databases or any critical service in production.
  • Use named volumes for persistent data — avoid bind mounts for database containers.
  • Set restart policies — restart: unless-stopped keeps services up after host reboots.
  • Use .env files for secrets — never commit passwords to version control inside compose files.
  • Remove the version: key from Compose files — it is obsolete in Compose v2.
  • Run docker system prune regularly on long-lived hosts to reclaim disk space.
  • Check docker logs immediately when a container exits — the error is almost always there.
  • For Postgres 18+, mount volumes at /var/lib/postgresql (not the /data subdirectory).
  • Do not add users to the docker group casually — it is equivalent to granting sudo.
  • Use docker compose config to validate compose files before deploying changes.
