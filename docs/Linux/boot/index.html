Linux Boot Process

The Linux boot process is a complex sequence of events that transforms a powered-off computer into a fully functional operating system ready for user interaction. This document provides a detailed exploration of each stage, from the initial electrical signals to the final userspace initialization.

Stage 1: Power-On and Hardware Initialization

Pressing the Power Button

When you press the power button, you complete an electrical circuit that signals the power supply unit (PSU) to begin operation. The PSU performs a Power-On Self-Test (POST) and, if successful, sends a “power good” signal to the motherboard. This signal indicates that stable power is available to all components.

Reset Vector and CPU Initialization

Upon receiving power, the CPU is held in a reset state. When released from reset, the CPU’s instruction pointer is set to a predefined memory address called the reset vector. On x86/x86_64 architectures, this address is typically 0xFFFFFFF0, which is near the top of the 4GB address space and maps to the firmware ROM.

The CPU begins executing in real mode (16-bit mode), a legacy compatibility mode that provides access to only 1MB of memory. At this point, no operating system code has loaded—only the firmware is accessible.

Stage 2: Firmware Execution (BIOS/UEFI)

BIOS (Basic Input/Output System)

Traditional BIOS firmware performs several critical functions:

Power-On Self-Test (POST): The BIOS executes diagnostic tests on system hardware including:

  • Memory integrity checks
  • CPU verification
  • Keyboard and peripheral detection
  • Video card initialization
  • Storage device enumeration

Hardware Initialization: The BIOS initializes and configures hardware components, setting up interrupt vectors, configuring the Programmable Interrupt Controller (PIC), and establishing basic I/O operations.

Boot Device Selection: The BIOS reads the boot order configuration from CMOS memory and attempts to locate a bootable device. It searches for a valid Master Boot Record (MBR) by reading the first 512 bytes (sector 0) of each configured boot device.

MBR Loading: Once a valid MBR is found (identified by the boot signature 0x55AA in bytes 510-511), the BIOS loads this 512-byte sector into memory at address 0x7C00 and transfers execution to it.

UEFI (Unified Extensible Firmware Interface)

Modern UEFI firmware provides enhanced capabilities:

Initialization: UEFI performs similar POST operations but with a more sophisticated architecture. It operates in 32-bit or 64-bit mode (not limited to 16-bit real mode) and provides a more extensive pre-boot environment.

GPT Support: UEFI natively supports the GUID Partition Table (GPT) scheme, which overcomes MBR limitations (2TB disk size limit, maximum 4 primary partitions).

Boot Manager: UEFI includes a built-in boot manager that reads boot entries from NVRAM. Each entry points to an EFI application (typically a bootloader) stored in the EFI System Partition (ESP), a FAT32-formatted partition.

Secure Boot: UEFI can verify cryptographic signatures of bootloaders and kernel images, preventing unauthorized code execution during boot.

Stage 3: Bootloader Execution

GRUB (Grand Unified Bootloader)

GRUB is the most common bootloader for Linux systems and operates in multiple stages:

Stage 1 (boot.img): In BIOS systems, this 512-byte code resides in the MBR. Its sole purpose is to load Stage 1.5 or Stage 2. In UEFI systems, the firmware directly loads the GRUB EFI application from the ESP.

Stage 1.5 (core.img): This intermediate stage contains filesystem drivers. It’s typically stored in the space between the MBR and the first partition (MBR gap) or in a dedicated boot partition. Stage 1.5 provides the ability to read filesystems like ext4, XFS, or Btrfs.

Stage 2: The full GRUB environment loads, presenting the boot menu interface. GRUB reads its configuration from /boot/grub/grub.cfg (or /boot/grub2/grub.cfg), which contains:

  • Kernel image locations
  • Initial ramdisk (initrd/initramfs) locations
  • Kernel command-line parameters
  • Boot menu entries and timeout settings

Kernel Loading: When a menu entry is selected, GRUB:

  • Loads the compressed kernel image (typically /boot/vmlinuz-) into memory
  • Loads the initial RAM filesystem (initramfs/initrd) into memory
  • Passes control to the kernel with specified command-line parameters

Kernel Command-Line Parameters

Common parameters include:

  • root=/dev/sda1 or root=UUID=…: Specifies the root filesystem location
  • ro: Mounts root filesystem as read-only initially
  • quiet: Suppresses verbose boot messages
  • splash: Enables graphical boot splash screen
  • init=/sbin/init: Specifies the init system path
  • systemd.unit=rescue.target: Boots into a specific target

Stage 4: Kernel Initialization

Kernel Decompression and Setup

The kernel image is compressed (using gzip, bzip2, or xz) to reduce size. The kernel contains a small decompression stub that:

  • Decompresses the kernel into memory
  • Performs relocation if necessary
  • Jumps to the kernel’s entry point

Early Kernel Initialization (start_kernel)

The start_kernel() function in init/main.c is the kernel’s main entry point. It performs initialization in a specific order:

Architecture-Specific Setup: Calls setup_arch() to initialize CPU-specific features, memory management structures, and hardware detection.

Memory Management Initialization:

  • Sets up page tables and enables paging
  • Initializes the buddy allocator for physical memory management
  • Configures memory zones (DMA, Normal, HighMem)
  • Establishes the slab allocator for kernel object caching

Trap and Interrupt Setup:

  • Configures the Interrupt Descriptor Table (IDT)
  • Sets up exception handlers for CPU faults
  • Initializes the interrupt controller (APIC on modern systems)

Scheduler Initialization: Configures the process scheduler, creating data structures for runqueues and initializing the first kernel thread (PID 0, the idle task).

Time Management: Initializes system timers, clocksources, and the tick mechanism for scheduling and timekeeping.

Console Initialization: Sets up early console output for kernel messages, enabling printk() output to be visible. Subsystem Initialization

The kernel initializes various subsystems through a series of initcall functions organized by priority levels:

Core Subsystems:

  • Virtual File System (VFS) layer
  • Process management structures
  • Signal handling mechanisms
  • Inter-Process Communication (IPC) facilities

Device Model: Initializes the kernel’s device model, including:

Sysfs filesystem mounting Device class registration Bus type registration (PCI, USB, etc.)

Driver Initialization: Loads built-in drivers in dependency order:

  • Bus drivers (PCI, USB controllers)
  • Storage controllers (SATA, NVMe, SCSI)
  • Network interface drivers
  • Input device drivers
  • Graphics drivers

Initial RAM Filesystem (initramfs)

The initramfs is a compressed cpio archive containing a minimal root filesystem. The kernel extracts it into a tmpfs filesystem mounted as the initial root. The initramfs typically contains:

Essential Binaries: Minimal shell, basic utilities (often from busybox)

Kernel Modules: Drivers needed to access the real root filesystem (filesystem drivers, RAID modules, encryption modules, device mapper)

Init Scripts: Scripts that perform early userspace tasks:

  • Loading necessary kernel modules
  • Assembling software RAID arrays
  • Unlocking encrypted volumes (LUKS)
  • Configuring network interfaces for network boot
  • Mounting the real root filesystem

udev/systemd-udevd: Device manager for dynamic device node creation

Mounting the Real Root Filesystem

The initramfs init script performs the following:

  • Module Loading: Uses modprobe to load required drivers based on detected hardware
  • Device Detection: Waits for the root device to appear (using udev)
  • Filesystem Check: Optionally runs fsck on the root filesystem
  • Root Mount: Mounts the real root filesystem (specified by the root= kernel parameter) at /sysroot or similar mountpoint
  • Switch Root: Uses switch_root or pivot_root to change the root filesystem from initramfs to the real root, then executes the real init system
