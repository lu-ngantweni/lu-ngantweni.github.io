Connecting Ansible over AWS Session Manager

15 Mar 2023 in posts

How to configure Ansible to connect over AWS Session Manager

This article aims to explain, how to configure ansible to connect over SSM session manager connection and execute playbooks.

  1. 1. Install aws collections using ansible-galaxy
  2. 2. Enable the plugin in ansible.cfg
  3. 3. Create inventory file
  4. 4. Create and execute a playbook
  5. Troubleshooting
    1. Sample log output from cloudwatch logs:

The purpose of this article is to explain how to execute Ansible playbooks on EC2 instances over SSM Session Manager connection. To achieve this use-case we make use of Ansible Collections, these are a distribution format for Ansible content that can include playbooks, roles, modules, and plugins. For this exercise we will make use of the following plugins:

amazon.aws.aws_ec2 - Get inventory hosts from EC2
community.aws.aws_ssm - This connection plugin allows Ansible to execute tasks on an EC2 instance via the aws ssm CLI

The above plugins form part of the amazon.aws (maintained by the Ansible cloud team) and community.aws (maintained by the Ansible community) collections. These modules and plugins offer a variety of Ansible content to help automate the management of AWS resources.

These are third-party plugins and customers can reach out to vendor and/or community for support issues pertaining these plugins. We can however assist with SSM connection issues and session manager plugin. For everything else, best efforts.

Okay! Let’s put the lab together by preparing our instances.

Controller Instance:
Ansible
SSM Agent
Session manager plugin
python >= 3.6
boto3 >= 1.16.0
awscli (optional)

Target Instance (s) SSM Agent curl installed

Once we’ve prepped our environment, our next steps will be to configure our inventory using the amazon.aws.aws_ec2

1. Install aws collections using ansible-galaxy

ansible-galaxy collection install community.aws

When you install the community.aws collection it will automatically install the amazon.aws collection.

2. Enable the plugin in ansible.cfg

This can be done on global config /etc/ansible/ansible.cfg, for this exercise we will create a local cfg file.

First we create a project folder and cd into that folder:

mkdir project1
cd project1

Generate ansible.cfg file:

ansible-config init --disabled > ansible.cfg

Using your favorite text editor, edit ansible.cfg » find entry: [inventory] and enable EC2 plugin, similar to below:

# file: "ansible.cfg"
[inventory] enable_plugins = amazon.aws.aws_ec2

Once the above is complete, you should now have a ansible.cfg file with plugin enabled for local testing. Next steps will be to create inventory.

3. Create inventory file

To create inventory file, we create aws_ec2.yml in our project directory (Use a YAML configuration file that ends with aws_ec2.{yml yaml}). 
sudo vi aws_ec2.yml

Here we specify our ec2_inventory plugin and set some parameters, can reference docs for parameter options.

# file: "aws_ec2.yml"
##Inventory
plugin: amazon.aws.aws_ec2
regions:
 - "us-east-1"
filters:
 instance-state-name : running
hostnames:
 - instance-id
compose:
  ansible_host: instance-id

Now that we have our inventory file, let’s do a test run by performing a list operation:

{
  "all": {
    "children": [
      "aws_ec2",
      "ungrouped"
    ]
  },
  "aws_ec2": {
    "hosts": [
      "i-04dc05dca9b5ede89",
      "i-0e6ee2fa57c6d1b46"
    ]
  }
}

Inventory test output showing 2 discovered instances.

As we can see from the truncated output, ec2-inventory returned 2 instances (Controller and target) now that we confirmed that this works, lets execute a playbook.

4. Create and execute a playbook

Using your text editor of choice, let’s create a playbook to install nginx. You can reference docs for variable options and for example playbooks for both windows and linux.

# file: "test.yml"
---
- name: Install a Nginx Package on Linux Instance
  hosts: all
  vars:
    ansible_connection: community.aws.aws_ssm
    ansible_aws_ssm_region: us-east-1
    ansible_aws_ssm_bucket_name: testing-inspector-policy
  tasks:
    - name: Install a Nginx Package
      yum:
        name: nginx
        state: present

Using the –limit option will exclude controller from play.

To confirm if the nginx was installed on instance, we can connect via session manager and check if the service is present (inactive):

aws ssm start-session --target i-0e6ee2fa57c6d1b46
Starting session with SessionId: testbench_DRS-024b6e23d7767846e

sh-4.4$ sudo systemctl status nginx
● nginx.service - The nginx HTTP and reverse proxy server
   Loaded: loaded (/usr/lib/systemd/system/nginx.service; disabled; vendor preset: disabled)
   Active: inactive (dead)

As expected service is installed but not running due to “present” flag.

Troubleshooting

Before working with customer to address issues set the correct expectations regarding support for third-party tools/applications.

Ansible will establish connection to ec2 instance via Session Manager, we start by verifying if we can connect to the instance using session manager without all the bells and whistles:

aws ssm start-session --target i-023500a698884d434
Starting session with SessionId: lunding_awscli-0ef41eeb1b8cc2296
sh-4.4$

Confirm that session starts without error and run a few commands and confirm that there is no issue with session duration. If you happen to run into any errors you can always review the following documentation for troubleshooting.

If you able to connect to the instance with session manager from terminal, we can rule out SSM. Troubleshooting beyond this point will be on best efforts.

Check for any syntax errors in the following files [ansible.cfg, playbook, inventory, etc]

If the above checkouts, rerun the play using -v, –vv or –vvv depending on the level of verbosity to catch any errors from an ansible perspective.

ansible-playbook -i aws_ec2.yml test.yml -vvv --limit '!i-0b52f059504ba450e'

The verbose output provides detailed information about each step of the playbook execution including connection establishment, module execution, and task results. Review the EXEC commands for an overview of commands executed on the instance.

  • The outputs of the commands can also be reviewed via cloudwatch logs, for more information about saving session data and other session manager preferences you can review this article

Sample log output from cloudwatch logs:

CloudWatch log output showing session activity.

Should you get any errors on logs you can assist with best efforts, alternatively customers can open an issue amazon.aws or community.aws depending on the issue experienced. Once connection is confirmed to be working from a Session Manager perspective all should be good from an SSM perspective, Alternatively customers can use RunCommand to execute ansible playbooks.
