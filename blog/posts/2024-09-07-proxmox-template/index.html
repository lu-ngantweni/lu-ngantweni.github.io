Preparing Proxmox Template

07 Sep 2024 in posts

This post covers a basic walkthrough on steps I generally take to prepare proxmox VM for template

A template is a fully pre-configured operating system image that can used to deploy KVM virtual machines. Creating a special template is usually preferred over cloning an existing VM. Deploying virtual machines from templates is blazing fast, very comfortable and if you use linked clones you can optimize your storage by using base images and thin-provisioning.

Creating a template will automatically convert your current VM to a template, to ensure that every VM launched from the template is unique there are configurations which can be made to ensure that VMs launched from template are consistent and unique

1. Perfrom an system update

This is to ensure that your newly launched VM is launched with latest packages, this step is also generally required when launching and accessing new VM to ensure that your system is on latest available packages.

# file: "system-update.sh"
sudo apt update && sudo apt upgrade -y

2. Create passwordless sudo

Passwordless sudo just makes managing VM tasks much easier, you have the option of running commands as root but this creates files and folders as root. Leaning towards security best practices using sudo just makes system more secure and passwordless sudo makes running priviledged commands much easier as it won’t prompt for password.

# file: "visudo-config"
sudo vi visudo

# Add the following entry (replace with username)
sysadmin ALL=(ALL) NOPASSWD:ALL

OR

#Add the rule for all members of wheel group
%wheel ALL = (ALL) NOPASSWD: ALL

3. Install cloud-init

Install needed utilities, in this command we installing cloud-init, a tool that automates the initialization of cloud and VM instances. You can configure cloud-init to perform a variety of tasks during boot, will be covered in later chapters.

sudo apt install cloud-init

4. Confirm and clear machine-id

The machine ID is usually generated from a random source during system installation or first boot and stays constant for all subsequent boots. Optionally, for stateless systems, it is generated during runtime during early boot if necessary.

sudo cat /etc/machine-id
sudo truncate -s 0 /etc/machine-id

5. Confirm if the machine id symbolic link exists, if not you can create one

The /var/lib/dbus/machine-id is generally the directory used by dbus to store the UUID for the machine-ID, by creating the symbolic link we are ensuring that there is consistency between the two locations. This symbolic link should be there by default and creating it is optional.

sudo ls -l /var/lib/dbus/machine-id
sudo ln -s /etc/machine-id /var/lib/dbus/machine-id

6. Remove ssh_host keys

Technically, manually removal of the ssh_host keys are not needed as that is done by the script in step.7 as well, prior to obtaining the script I manually removed the keys with the hopes of having them created by cloud-init that does not always go well….

sudo cd /etc/ssh
sudo rm ssh_host_*

7. Us the below text script to create one shot service to regenerate hostkeys (copy/paste below to regenerate_ssh_host_keys.service)

Credit for this script is Jay from learnLinux TV and can be found here.

# file: "regenerate_ssh_host_keys.service"
[Unit]
Description=Regenerate SSH host keys
Before=ssh.service
ConditionFileIsExecutable=/usr/bin/ssh-keygen

[Service]
Type=oneshot
ExecStartPre=-/bin/dd if=/dev/hwrng of=/dev/urandom count=1 bs=4096
ExecStartPre=-/bin/sh -c "/bin/rm -f -v /etc/ssh/ssh_host_*_key*"
ExecStart=/usr/bin/ssh-keygen -A -v
ExecStartPost=/bin/systemctl disable regenerate_ssh_host_keys

[Install]
WantedBy=multi-user.target
Follow these steps to create systemd files

The below steps covers general guidance on creating a systemd unit file.

sudo vi regenerate_ssh_host_keys.service
sudo chown root:root regenerate_ssh_host_keys.service
sudo mv regenerate_ssh_host_keys.service /etc/systemd/system
sudo systemctl daemon-reload
sudo systemctl enable regenerate_ssh_host_keys.service

8. Perform a repo clean and remove stale applications

sudo apt clean
sudo apt autoremove

9. Clear the history file

history -c && history -w
